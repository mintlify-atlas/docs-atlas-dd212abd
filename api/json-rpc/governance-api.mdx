---
title: Governance API Methods
description: JSON-RPC methods for accessing staking, validator, and governance data
---

The Governance API provides access to validator and staking-related data such as current committee info, delegated stakes, and APY. All methods are in the `iotax` namespace.

## iotax_getStakes

Return all DelegatedStake objects for an address.

<ParamField path="owner" type="IotaAddress" required>
  The owner's IOTA address
</ParamField>

<ResponseField name="stakes" type="Vec<DelegatedStake>">
  List of delegated stakes
  
  <Expandable title="DelegatedStake">
    <ResponseField name="stakedIotaId" type="ObjectID">
      The object ID of the staked IOTA
    </ResponseField>
    <ResponseField name="stakeRequestEpoch" type="BigInt<u64>">
      The epoch when the stake was requested
    </ResponseField>
    <ResponseField name="stakeActiveEpoch" type="BigInt<u64>">
      The epoch when the stake becomes active
    </ResponseField>
    <ResponseField name="principal" type="BigInt<u64>">
      The principal amount staked
    </ResponseField>
    <ResponseField name="status" type="String">
      Status: "Active", "Pending", or "Unstaked"
    </ResponseField>
    <ResponseField name="estimatedReward" type="BigInt<u64>">
      Estimated reward for active stakes
    </ResponseField>
  </Expandable>
</ResponseField>

```bash cURL
curl -X POST https://api.iota.io \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "iotax_getStakes",
    "params": [
      "0x0479c602460ae68771dcb9bfcef607dccc678859fb72d7d8aa8d9ce58c9430747"
    ]
  }'
```

## iotax_getCommitteeInfo

Return the committee information for the asked epoch.

<ParamField path="epoch" type="BigInt<u64>">
  The epoch of interest. If None, defaults to the latest epoch
</ParamField>

<ResponseField name="epoch" type="BigInt<u64>">
  The epoch number
</ResponseField>

<ResponseField name="validators" type="Vec<CommitteeValidator>">
  List of validators in the committee
  
  <Expandable title="CommitteeValidator">
    <ResponseField name="authorityName" type="String">
      The validator's name
    </ResponseField>
    <ResponseField name="iotaAddress" type="IotaAddress">
      The validator's IOTA address
    </ResponseField>
    <ResponseField name="stakingPoolId" type="ObjectID">
      The staking pool object ID
    </ResponseField>
    <ResponseField name="votingPower" type="BigInt<u64>">
      The validator's voting power
    </ResponseField>
  </Expandable>
</ResponseField>

```bash cURL
curl -X POST https://api.iota.io \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "iotax_getCommitteeInfo",
    "params": [null]
  }'
```

## Staking Workflow

Typical staking operations involve:

1. **Query Validators** - Use `iotax_getLatestIotaSystemStateV2` to get validator information
2. **Check APY** - Use `iotax_getValidatorsApy` to compare validator rewards
3. **Stake IOTA** - Build and execute a staking transaction
4. **Monitor Stakes** - Use `iotax_getStakes` to track your delegated stakes
5. **Unstake** - Build and execute an unstaking transaction when desired

## Understanding Epochs

IOTA's staking system operates on epochs:

- Staking requests become active in the next epoch
- Unstaking requests are processed at epoch boundaries
- Rewards are calculated and distributed per epoch
- Use `iotax_getCommitteeInfo` to query historical epoch data
